# 갈만한 컨퍼런스·밋업 찾기

Luma와 Meetup에서 다가오는 이벤트를 모아, 사용자의 관심 프로필에 맞는 것을 관련도순으로 추천한다.

## 관심 프로필 (기본값)

아래를 관련도 평가의 기본 기준으로 삼는다. 사용자가 실행 시 "이번엔 백엔드만", "무료만", "주말만" 식으로 좁히면 그 조건을 우선한다.

- **AI** (전반)
- **AI 에이전트 / LLM 애플리케이션**
- **백엔드 개발**
- **신기술**
- **개발자 네트워킹**
- **스타트업 / 빌더 / 해커톤**

"가서 인사이트를 많이 얻을 수 있는" 자리를 선호한다 → 단순 홍보성 행사보다, 실무 발표·핸즈온 워크숍·깊이 있는 네트워킹이 있는 이벤트에 가산점을 준다.

## 진행 순서

체크리스트로 만들어 순서대로 진행한다.

### 1. Luma 이벤트 읽기 (1차 소스 — 공개 JSON API 우선)

- Luma는 **인증 불필요 공개 JSON API**로 이벤트를 **날짜(`start_at`)까지** 깨끗하게 준다. HTML 페이지는 JS 렌더링이라 날짜가 안 잡히므로, **API를 먼저** `WebFetch`로 읽는다:
  - **AI 카테고리**: `https://api.luma.com/discover/get-paginated-events?category_slug=ai&place_slug=seoul`
  - **도시 전체**: `https://api.luma.com/discover/get-paginated-events?slug=seoul`
  - 각 이벤트에서 이름(`name`), `start_at`, slug(`url`)을 뽑는다.
  - ⚠️ `start_at`은 **UTC(…Z)**라 **+9시간 해서 KST(한국 시간)로 변환**한다 (예: `2026-08-01T04:30:00Z` → 8/1 13:30 KST).
  - 개별 이벤트 링크는 `https://luma.com/<slug>`.
- 폴백: API 결과가 부실하면 HTML 페이지 `https://luma.com/discover/seoul/ai`·`/tech`·`/seoul`을 `WebFetch`로 읽는다 (이 경우 날짜는 4.5단계에서 처리).
- 다른 도시: `place_slug`/`slug`를 바꾼다 (예: `place_slug=busan`).

### 2. Meetup 검색 읽기 (2차 소스)

- 관심 키워드별로 `WebFetch`로 Meetup 검색 페이지를 읽는다. URL 형식:
  `https://www.meetup.com/find/?keywords=<키워드>&location=kr--11--Seoul&source=EVENTS`
- 키워드는 관심 프로필에서 1~3개를 골라 각각 조회한다 (예: `AI`, `backend`, `DevOps`). Meetup은 날짜가 잘 나오므로 날짜 정보를 여기서 확보한다.

### 3. 웹검색 + Dev-Event 읽기 (3차 소스 — 큰 컨퍼런스)

Luma·Meetup은 소규모 밋업 위주라 **대형 컨퍼런스를 놓친다.** 이를 정식 소스로 보완한다.

- **웹검색 (`WebSearch`)**: 관심사·시기로 검색해 큰 컨퍼런스를 찾는다 (예: "서울 AI 컨퍼런스 2026", "Seoul developer conference 2026"). 밋업=작은 모임, 웹검색=큰 컨퍼런스로 서로 보완한다. (WebSearch는 US 기준이라 한국 이벤트엔 약할 수 있으니, 한국어·영어 쿼리를 함께 시도한다.)
- **Dev-Event 목록 (`WebFetch`)**: 한국 개발자 행사를 큐레이션한 저장소 `https://github.com/brave-people/Dev-Event` 를 읽어 다가오는 컨퍼런스·해커톤·웨비나를 추출한다. (README 또는 해당 월 파일)
- 결과가 부실하면 조용히 넘어간다. 여기서 얻은 큰 컨퍼런스도 다음 단계의 순위화 대상에 포함한다.

### 4. 관련도 평가·순위화 (중복 제거)

- Luma·Meetup·웹검색·Dev-Event 결과를 모두 합치고, 같은 이벤트는 하나로 합친다.
- 각 이벤트를 관심 프로필과 대조해 **관련도(상/중/하)**를 매기고, "인사이트 가치"가 높아 보이는 순으로 정렬한다.
- 사용자가 좁힌 조건(주제·무료·요일·지역)이 있으면 그걸로 먼저 거른다.

### 4.5. 날짜 확인 (대부분 자동)

- Luma JSON API(Step 1)가 `start_at`을 주므로 **날짜는 대개 이미 확보**된다. UTC→KST(+9) 변환만 확인한다.
- 그래도 날짜가 비어 있는 이벤트는:
  1. 개별 페이지(`https://luma.com/<slug>`)를 `WebFetch`로 한 번 시도한다.
  2. **그래도 안 나오면 지어내지 말고 사용자에게 물어본다** — "이 이벤트 날짜가 안 잡혔어요. 페이지에서 확인해 알려주실래요?" (로그인 상태면 브라우저엔 날짜가 보인다.)
- Meetup은 검색 목록에 날짜가 잘 나오므로 확인이 대개 불필요하다.

### 5. 추천 목록 제시

**개수 제한 없이, 관심 프로필에 조금이라도 맞는 이벤트는 최대한 다 보여준다** (관련도 높은 순으로 정렬). 관련도 `하`까지 포함하되, 프로필과 전혀 무관한 것(소개팅·와인·콘서트 등)만 제외한다.

**먼저 한눈 요약 표**를 맨 위에 제시한다 (전체를 스크롤 없이 스캔하도록):

| # | 이벤트 | 유형 | 날짜 | 관련도 |
|---|--------|------|------|--------|
| 1 | (제목 간략) | 핸즈온/컨퍼런스/네트워킹/해커톤 등 | 날짜 or 미확인 | 상/중/하 |

- "유형"은 무슨 이벤트인지 한 단어로 분류한다 (핸즈온 워크숍 / 대형 컨퍼런스 / 밋업·네트워킹 / 해커톤 / 세미나·강연 등).
- 표로 전체를 먼저 훑게 한 뒤, 아래에 상세 카드를 이어서 보여준다.
- 개수가 많으면(15개+) 관련도 `상`·`중`은 상세 카드까지, `하`는 표에만 간단히 넣어도 된다.

**그다음 상세 카드**를 아래 형식으로 보여준다:
  - **제목** — 관련도(상/중/하), 날짜(있으면), 장소, 주최자, 상태
  - **출처**: 어느 플랫폼에서 찾았는지(Luma / Meetup / 웹검색)와 **링크를 함께 표기**한다.
    - 링크는 반드시 **마크다운 하이퍼링크** 문법으로 출력한다: `[신청/상세 페이지](https://...)`. 날 URL을 괄호로 노출하지 말고 클릭 가능한 링크로 만든다.
    - 개별 이벤트 URL을 확보했으면 그 URL로 링크한다 (예: `[이벤트 페이지](https://luma.com/307r8yon)`).
    - 개별 URL이 목록에서 안 잡혔으면(Luma 서울 페이지 등), 최소한 **찾은 출처 목록 페이지**로 링크하고(예: `[Luma 서울](https://luma.com/seoul)`) "개별 링크는 상세 확인 단계에서 찾는다"고 밝힌다.
  - **무슨 이벤트** 한 줄 요약 — 이 행사가 실제로 뭘 하는 자리인지(주제·형식: 강연/워크숍/네트워킹 등)를 확인된 정보로 요약한다. 제목만으로 불분명하면 개별 페이지를 `WebFetch`해 파악하고, 그래도 모르면 지어내지 말고 "상세 미확인"으로 둔다.
  - **추천 이유** 한 줄 (어떤 관심사에 맞는지, 왜 인사이트가 있을지)
- 목록 끝에 "더 자세히 보고 싶은 이벤트를 골라주세요"라고 안내한다.

### 6. 상세 확인

- 사용자가 고른 이벤트의 **개별 페이지 URL**을 `WebFetch`로 열어 정확한 날짜·시간·장소·신청 링크·상세 내용을 확인해 알려준다.
- 개별 페이지 URL을 모르면, 제목으로 `WebSearch`하거나 사용자에게 링크를 물어본다.

### 7. Notion 기록 (선택)

추천 결과를 **Notion "컨퍼런스 트래커" DB**에 저장한다.

- 대상 DB: `컨퍼런스 트래커`
  - 페이지 URL: `https://app.notion.com/p/80fee3dffbbd436d8e5161b1fce0410f`
  - 데이터 소스: `collection://6edf60d6-31eb-4b30-9836-9012b6fd8502`
- **먼저 물어본다**: "이 결과를 Notion 컨퍼런스 트래커에 저장할까요? (전체 / 고른 것만 / 안 함)". 사용자가 원할 때만 저장한다.
- `mcp__claude_ai_Notion__notion-create-pages`로 위 데이터 소스에 각 이벤트를 한 행씩 추가한다. 속성 매핑:
  - `이벤트명`=제목, `유형`=핸즈온/컨퍼런스/밋업/해커톤/세미나 중 하나, `날짜`=확인된 날짜(없으면 비움), `장소`, `관련도`=상/중/하, `출처`=Luma/Meetup/웹검색/Dev-Event, `링크`=신청/상세 URL, `요약`=Step 5의 "무슨 이벤트" 한 줄 요약, `상태`=`가고싶음`(기본)
- **중복 방지**: 저장 전에 `mcp__claude_ai_Notion__notion-search`로 같은 이벤트명이 이미 있는지 확인하고, 있으면 건너뛴다.
- 저장 후 추가된 개수와 DB 링크를 알려준다.

## 주의사항

- 이벤트 정보를 지어내지 않는다. WebFetch·WebSearch로 확인된 것만 쓰고, 날짜·링크가 불확실하면 "목록에서 날짜가 확인되지 않음"이라고 솔직히 밝힌다.
- 매진·마감된 이벤트는 상태를 함께 표기해 사용자가 헛걸음하지 않게 한다.
- 다녀온 뒤 후기를 쓰고 싶어 하면 `conference-recap` 스킬로 이어준다.
